# AUTH SECURITY AUDIT

**Date:** 2026-06-10
**Scope:** Review of the custom database‑only authentication implementation introduced in the ER Tracker project.

## 1. Password Hashing Algorithm
* **Algorithm:** PostgreSQL `crypt` with the **bcrypt** (`bf`) method.
* **Implementation:** In `DATABASE_AUTH_MIGRATION.sql` the `password_hash` column stores the result of `crypt(<plain>, gen_salt('bf'))`.
* **Verification:** Password verification is performed via `crypt(provided_password, stored_hash) = stored_hash` in the login query.
* **Security Rating:** ✅ Strong – bcrypt is a proven, adaptive hash function with configurable work factor.

## 2. Session Token Generation
* **Method:** UUID v4 generated by PostgreSQL `uuid_generate_v4()`.
* **Location:** Inserted into `public.user_sessions.session_token` upon successful login.
* **Entropy:** UUID v4 provides ~122 bits of randomness, sufficient for session tokens.
* **Security Rating:** ✅ Good – unpredictable, collision‑resistant.

## 3. Session Token Storage
* **Server‑Side:** Stored in `public.user_sessions` with a FK to the user.
* **Client‑Side:** Expected to be returned to the client and stored in an **HttpOnly cookie** or secure local storage (implementation pending in frontend).
* **Best‑Practice Note:** HttpOnly cookies mitigate XSS theft; ensure `Secure` flag is set when served over HTTPS.

## 4. Session Expiration
* **Column:** `expires_at` in `user_sessions` defines the expiry timestamp.
* **Default Policy:** Migration does not set a default; application code should set a sensible TTL (e.g., now() + interval '7 days').
* **Enforcement:** Each request must validate that `expires_at > now()` before accepting the session.
* **Security Rating:** ⚠️ Needs implementation – ensure expiry is applied and refreshed on activity.

## 5. Route Protection
* **Current State:** No explicit API layer exists yet for the custom auth; route protection is still handled by Supabase RLS for existing tables.
* **Recommendation:** Add middleware (e.g., Express/Node or FastAPI) that:
  1. Extracts the session token.
  2. Looks up `user_sessions` and verifies expiry.
  3. Attaches the user ID and role to the request context.
  4. Enforces role‑based access on protected endpoints.

## 6. Password Change Flow
* **Planned Implementation:** An endpoint should accept the current password, verify it using `crypt`, then update `password_hash` with a new bcrypt hash.
* **Audit:** Ensure the change is logged in `audit_logs` with action `PASSWORD_CHANGE`.
* **Security Rating:** ⚠️ Not yet implemented – must be added.

## 7. User Disable Enforcement
* **Column:** `status` in `users` (default `active`).
* **Enforcement:** Application login logic must check `status = 'active'` before allowing session creation.
* **Audit:** Disabled logins should be recorded as `LOGIN_FAILURE` with reason.
* **Security Rating:** ⚠️ Partial – status check needs to be added to login flow.

## Recommendations & Action Items
1. **Implement middleware** for session validation and role enforcement.
2. **Add expiry handling** on session creation (e.g., 7‑day TTL) and refresh logic.
3. **Store session token** in HttpOnly, Secure cookie on the client.
4. **Create password‑change endpoint** with proper verification and audit logging.
5. **Enforce user status** check during login and reject disabled accounts.
6. **Update documentation** in `DATABASE_AUTH_ARCHITECTURE.md` to reflect these controls.
