# PROJECT ARCHITECTURE

## 1. Project Purpose

The **ER Tracker Dashboard** is a web‑based administration console for managing users, roles, and workbooks in a Supabase‑backed PostgreSQL database. It provides a single‑page application (SPA) built with React and TypeScript, hosted on Vercel.

## 2. Current Architecture

| Layer | Technology | Details |
|-------|------------|---------|
| **Frontend** | React, TypeScript, Vite | UI components, routing, state management via React Context. All business logic lives in the browser.
| **Database** | Supabase PostgreSQL | Public schema with tables `users`, `roles`, `user_roles`, `workbooks`, `workbook_rows`, etc.
| **Authentication** | Custom – localStorage session management | No Supabase Auth, no FastAPI, no Edge Functions (for now). Passwords are hashed in the browser before persisting.
| **Deployment** | Vercel (static build) | The `dist/` folder generated by Vite is deployed; the Supabase database is managed separately via the Supabase dashboard.

## 3. Authentication Strategy

* **Create User** – Admin creates a user; the password is hashed in the browser using a client‑side library (`bcryptjs`). The hash is stored in `public.users.hashed_password`.
* **Login** – Username and password are supplied; the stored hash is fetched and verified. On success, the user object and assigned roles are saved in `localStorage` and the auth context is populated.
* **Session Persistence** – On page refresh the `AuthContext` reads the stored data from `localStorage` and restores the session.
* **Logout** – Clears `localStorage` and redirects to the login page.

## 4. User Management Strategy

* **Create / Edit / Disable** – Admin forms send payloads directly to Supabase via the `supabase-js` client.
* **Role Assignment** – Inserts rows into `public.user_roles` linking a user ID to role IDs.
* **Diagnostics** – Console logs are added throughout the service layer to aid troubleshooting.

## 5. Role Management Strategy

Roles are stored in `public.roles` with a simple `id`/`name` schema. Permissions are not yet modelled; they will be added in future iterations.

## 6. Workbook Management Strategy

Workbooks and their rows are stored in `public.workbooks` and `public.workbook_rows`. The UI provides CRUD operations via the Supabase client.

## 7. Dashboard Structure

The app uses a **MainLayout** with a navigation sidebar. Pages include:
* Dashboard (overview)
* Users / User Management
* Role Management
* Workbook Management
* Login / Logout

## 8. Database Schema Summary

```
users (id, username, hashed_password, is_active, created_at, updated_at)
roles (id, name)
user_roles (user_id, role_id)
workbooks (id, title, owner_id, created_at)
workbook_rows (id, workbook_id, data, created_at)
```

## 9. Deployment Strategy

1. **Frontend** – `npm run build` creates a static bundle in `dist/` which is deployed to Vercel.
2. **Database** – Managed via Supabase dashboard; migrations are applied manually or via `supabase db push`.
3. **Environment Variables** – Only `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` are needed in the frontend (stored in `.env`).
